# Radikale Venstre resolutions demo

A small static prototype for browsing and searching Radikale Venstre landsmøde resolutions.

The project is deliberately simple:

- DOCX files are parsed locally with Python.
- The generated data is written to `public/resolutions.json`.
- The website is a static, dependency-free HTML/CSS/JavaScript app.
- The app can be hosted by any static web server.

## Project structure

```text
public/
  index.html          Static frontend
  style.css           Plain CSS styling
  app.js              Dependency-free browser app
  resolutions.json    Generated dataset used by the frontend
scripts/
  parse_docx.py       DOCX-to-JSON parser
  validate_data.py    Sanity checker for generated JSON
source/
  *.docx              Original resolution documents
years.json            Adoption and validity dates per year
schema.json           JSON schema for the canonical output format
```

## Python environment with uv

This project uses uv for Python dependency management.

The Python dependencies are declared in pyproject.toml. The lock file uv.lock should be committed, so the parser and validation scripts can be run with reproducible dependencies.

Create or update the local virtual environment:

```
uv sync
```

Run the parser and validation scripts:

```bash
uv run python scripts/parse_docx.py source/*.docx --out public/resolutions.json
uv run python scripts/validate_data.py public/resolutions.json
```

The `.venv/` directory is local developer state and should not be committed.

The generated `public/resolutions.json` file is not committed on the main branch. It is rebuilt from the Word documents and committed only to the pages branch as part of the published static site.

## Generate data

From the project root:

```bash
uv run python scripts/parse_docx.py source/*.docx --out public/resolutions.json
```

The parser reads `years.json` by default. Update that file before publishing new yearly data, especially if a date is currently marked as a placeholder.

The generated JSON uses this canonical shape:

```json
{
  "schema_version": 1,
  "generated_at": "...",
  "source_documents": [
    { "year": 2025, "filename": "...", "valid_from": "2025-09-21", "valid_until": "2030-09-21" }
  ],
  "resolutions": [
    { "id": "2025-A1", "title": "...", "body": "..." }
  ]
}
```

The frontend can still read the older raw-array format during prototyping, but new generated data should use the canonical wrapper.

## Validate data

Run the sanity checker after parsing:

```bash
uv run python scripts/validate_data.py public/resolutions.json
```

Optionally write a report to disk:

```bash
uv run python scripts/validate_data.py public/resolutions.json --report VALIDATION_REPORT.txt
```

Warnings do not always mean the data is wrong. They identify places worth reviewing, such as very short resolution bodies or possible page-number artefacts.

## Notes on data modelling

Local chapter codes such as `A`, `F` or `H` are treated as document-local codes only. The UI filters by chapter title/policy area because the same letter may not mean the same thing every year.

Keywords in the current prototype are generated by a simple deterministic Python method. They are useful for demos, but the intended long-term workflow is:

```text
DOCX -> raw parsed JSON -> LLM-generated search terms -> human review -> public JSON
```


## Policy area harmonisation

The source documents do not always use exactly the same chapter title for the same policy area. The parser therefore keeps the original `chapter_title`, but also writes a harmonised `policy_area` value used by the UI.

Policy area harmonisation is configured in `policy_areas.json`. Each canonical policy area has a list of exact aliases from the source documents. If the parser finds a new chapter title that is not listed, it will warn and continue using that title as its own policy area.

To add missing chapter titles as identity mappings, run:

```bash
uv run python scripts/parse_docx.py source/*.docx --out public/resolutions.json --update-policy-areas
```

Afterwards, review `policy_areas.json` manually. If a newly added area is just another wording of an existing area, move it into the existing area's `aliases` list and remove it as a separate canonical area. Then rebuild and validate the data.

## Manual smoke test

Run this test after changes to the parser, data format, or frontend. The purpose is not to proofread the resolution texts, but to check that the full prototype still works from Word documents to searchable user interface.

### 1. Rebuild data from scratch

```bash
rm -f public/resolutions.json
python scripts/parse_docx.py source/*.docx --out public/resolutions.json
python scripts/validate_data.py public/resolutions.json
python -m http.server 8080 -d public
```

Then open http://localhost:8080.


### 2. Check basic UI behaviour

The page loads without errors in the browser console.
The result counter shows a realistic number of resolutions.
The year filter shows 2023, 2024, and 2025.
The policy area filter shows relevant policy areas from the documents.
The "Only currently valid" filter changes the result list as expected.
A resolution can be opened and closed again.
The URL updates when searching and filtering, so a search can be shared.

### 3. Check a few known searches

Search for these terms and only check that the results look plausible:

open source
omvendelsesterapi
roadpricing
abort
Ukraine
tandlæge
monarkiet
PFAS

If a search returns surprisingly few or many results, that is not necessarily a frontend bug. It may also be caused by parser output, missing keywords, or the fact that the current search is still a simple substring/term-based search.

## Deploying to Codeberg Pages

The public demo is hosted as a static Codeberg Pages site.

The repository uses two branches:

- `main` contains the project source: parser scripts, source documents, metadata, schema, README, and the `public/` directory.
- `pages` contains only the generated static website files: `index.html`, `app.js`, `style.css`, `resolutions.json`, and any other files that should be served publicly.

For this repository, the expected public URL is:

```text
https://madsdyd.codeberg.page/rv-resolutioner/
```

### Codeberg Pages setup

The repository must have a `pages` branch, and Codeberg should have a Pages webhook configured for that branch.

In Codeberg, open the repository and go to:

```text
Settings -> Webhooks -> Add webhook -> Forgejo
```

Use this target URL:

```text
https://madsdyd.codeberg.page/rv-resolutioner/
```

Use this branch filter:

```text
pages
```

After the webhook has been created, pushing to the `pages` branch should trigger a new deployment.

### Manual deployment

From the `main` branch, rebuild and validate the static data:

```bash
python scripts/parse_docx.py source/*.docx --out public/resolutions.json
python scripts/validate_data.py public/resolutions.json
```

Then publish the contents of `public/` to the `pages` branch:

```bash
scripts/deploy_codeberg_pages.sh
```

The script creates a temporary worktree for the `pages` branch, copies the contents of `public/` into it, commits the changes if there are any, and pushes the `pages` branch.

### The important Git command

The actual command that publishes the site is:

```bash
git push origin pages
```

The deploy script runs that command from a temporary worktree where the checked-out branch is `pages`.

If deploying manually without the script, make sure you are pushing the `pages` branch, not `main`:

```bash
git switch pages
git push origin pages
```

Then switch back to the development branch:

```bash
git switch main
```

### Triggering a deployment without content changes

If the Codeberg Pages webhook needs to be triggered without changing files, create an empty commit on the `pages` branch:

```bash
git switch pages
git commit --allow-empty -m "Trigger Codeberg Pages deployment"
git push origin pages
git switch main
```

### Notes

- `public/resolutions.json` is generated data.
- The contents of the `pages` branch are public.
- The source `.docx` files do not need to be present on the `pages` branch.
- `index.html` must be at the root of the `pages` branch, not inside a `public/` directory.
- The frontend uses relative paths, so it should work both locally and under the repository path on Codeberg Pages.
